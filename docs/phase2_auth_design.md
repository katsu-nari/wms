## 1. 認証方式

### 採用方式: **社員番号 + 暗証番号（PIN）**

| 項目 | 内容 |
|------|------|
| 認証方式 | Supabase Auth — Email/Password |
| メールアドレス形式 | `{社員番号}@wms.internal`（疑似メール、実際には未送信） |
| パスワード | 4〜8桁の数字 PIN |
| ログイン画面 | 社員番号入力 + PIN入力の専用画面 |

### 他方式を採用しない理由

| 方式 | 不採用理由 |
|------|----------|
| Magic Link | 倉庫作業者が業務用メールを持っていないケースが多い。スマホ操作が煩雑 |
| Google/SNS OAuth | 個人アカウントとの混同リスク。社外サービス依存 |
| Phone OTP | SMS費用が発生。海外ワーカー対応が複雑 |
| LDAP / SAML | Supabase標準では非対応（Enterpriseプランが必要） |

### 疑似メール方式の説明

```
社員番号: 10042
↓ 変換
メールアドレス: 10042@wms.internal
パスワード: 1234 (PIN)
```

Supabase Auth の Email/Password 認証をそのまま利用できる。  
`@wms.internal` は実在しないドメインのため確認メールは不要（Supabase は Email Confirm を OFF に設定する）。

## 2. 認証フロー（技術構成）

### 実装アーキテクチャ

```
┌─────────────────────────────────┐
│  wms_supabase.html              │
│                                 │
│  ┌───────────────────────────┐  │
│  │  Supabase JS SDK (CDN)    │  │  ← 認証のみに使用
│  │  supabase.auth.signIn()   │  │
│  │  supabase.auth.signOut()  │  │
│  │  auto token refresh       │  │
│  └───────────────────────────┘  │
│              ↓                  │
│  H.Authorization = 'Bearer ' +  │
│    access_token                 │  ← ログイン後に上書き
│              ↓                  │
│  ┌───────────────────────────┐  │
│  │  既存 sb カスタムREST客     │  │  ← データ操作は変更なし
│  │  sb.get() / insert() 等   │  │
│  └───────────────────────────┘  │
└─────────────────────────────────┘
```

**方針**: Supabase JS SDK を CDN 経由で追加し、認証（ログイン・ログアウト・トークン更新）のみを担当させる。既存の `sb` カスタムRESTラッパーはデータ操作に引き続き使用し、ヘッダーの `Authorization` をログイン後に JWT に切り替える。

### ログインシーケンス

```
[ユーザー]                [ブラウザ]               [Supabase Auth]
    │                        │                           │
    │── 社員番号 + PIN ──────→│                           │
    │                        │── signInWithPassword ─────→│
    │                        │   email: 10042@wms.internal│
    │                        │   password: 1234           │
    │                        │                           │
    │                        │←── access_token ──────────│
    │                        │    refresh_token           │
    │                        │    user.id                 │
    │                        │                           │
    │                        │── profiles 取得 ──────────→│ (Supabase DB)
    │                        │←── { role, display_name }──│
    │                        │                           │
    │                        │   H.Authorization を更新   │
    │                        │   サイドバーにユーザー名表示│
    │←── ダッシュボード表示 ──│                           │
```

### ログアウトシーケンス

```
[ユーザー] ── ログアウトボタン ──→ signOut() ──→ localStorage クリア
                                              ──→ H.Authorization を anon に戻す
                                              ──→ S ステート初期化
                                              ──→ ログイン画面表示
```

## 3. 初期管理者の作成方法

### 手順

Step 2 実装後、以下の手順で最初の管理者アカウントを作成する。

**Step A: Supabase Dashboard で Email Confirm を無効化**

```
Dashboard → Authentication → Providers → Email
→ "Confirm email" を OFF に設定
```

**Step B: 管理者アカウント作成スクリプト（SQLエディタで実行）**

```sql
-- 1. auth.users にアカウントを作成（Supabase Dashboard 経由）
--    Authentication → Users → "Add user" ボタン
--    Email: admin001@wms.internal
--    Password: （管理者が設定）

-- 2. profiles レコードを挿入（auth.users 作成後に実行）
INSERT INTO profiles (id, employee_number, display_name, role)
SELECT id, 'admin001', '管理者', 'admin'
FROM auth.users
WHERE email = 'admin001@wms.internal';
```

**Step C: 動作確認**
ログイン画面から `admin001` + 設定したPINでログインできることを確認する。

## 4. 既存利用者の移行方法

### 現状

`wms_supabase.html` は認証なしで稼働しているため、**既存ユーザーアカウントは存在しない**。  
「移行」ではなく「新規全員登録」が必要になる。

### 登録方法（3パターン）

#### パターン A: 管理画面から1名ずつ登録（推奨：初期フェーズ）
管理者が Supabase Dashboard の Auth → Users から1名ずつ作成し、SQLで profiles を挿入する。

#### パターン B: CSVインポートスクリプト（一括登録）
```javascript

const users = [
  { emp: '10001', name: '山田 花子', role: 'operator' },
  { emp: '10002', name: '佐藤 一郎', role: 'viewer' },
  // ...
];
// supabase.auth.admin.createUser() で一括作成
```

#### パターン C: アプリ内ユーザー管理画面（Step 2+ の将来機能）
管理者ロールのユーザーが WMS 画面内でアカウントを作成できる機能。Step 2 本体の実装範囲外だが、将来的に追加を推奨する。

### 移行タイムライン（推奨）

## 5. profiles テーブル構成

### 既存スキーマ（0001_schema.sql — 変更不要）

```sql
create table if not exists profiles (
  id              uuid primary key references auth.users on delete cascade,
  employee_number text not null unique,        -- 社員番号（ログイン ID）
  display_name    text,                        -- 表示名（サイドバーに表示）
  role            text not null default 'viewer'
                  check (role in ('admin','operator','viewer')),
  is_locked       bool not null default false, -- アカウントロック
  locked_until    timestamptz,                 -- ロック解除日時
  failed_count    int  not null default 0,     -- 連続失敗回数
  last_login_at   timestamptz,                 -- 最終ログイン日時
  created_at      timestamptz not null default now(),
  updated_at      timestamptz not null default now()
);
```

**重要**: このテーブルはすでに migration `0001_schema.sql` で定義済みのため、Step 2 でスキーマ変更は不要。

### 各カラムの用途（Step 2 での使用箇所）

| カラム | Step 2 での用途 |
|--------|----------------|
| `id` | auth.users との紐付け |
| `employee_number` | ログインID（`@wms.internal` を除いた部分） |
| `display_name` | サイドバーのユーザー名表示（「田中 太郎」→実名に変更） |
| `role` | サイドバーのロール表示・将来の UI 出し分け |
| `is_locked` | ログイン時にチェック（ロック中は「アカウントがロックされています」表示） |
| `failed_count` | 5回失敗でアカウントロック（`login_attempts` と連動） |
| `last_login_at` | ログイン成功時に更新 |

## 6. ロール一覧

### DB ロール（3種類）と職位マッピング

現在の `profiles.role` は CHECK 制約で `admin / operator / viewer` の3値のみ許可している。  
運用上の職位区分（5種）は以下のように DB ロールにマッピングする。

| 職位（表示名） | DB ロール | 人数規模（目安） |
|--------------|----------|----------------|
| システム管理者 | `admin` | 1〜2名 |
| 倉庫管理者 | `admin` | 2〜5名 |
| 班長 | `operator` | 5〜10名 |
| 作業者 | `operator` | 50〜80名 |
| 荷主顧客 | `viewer` | 10〜20名 |

> **設計判断**: 職位区分を DB ロールに統合しているのは、既存の RLS ポリシー（Step 5 で有効化）が3値のみを前提としているため。  
> 将来的に班長と作業者を権限分離したい場合は `profiles` に `job_title` カラムを追加し、アプリ側でフィルタリングする設計を推奨する（RLS は3ロールのまま維持）。

## 9. ログアウト時の挙動

### 処理フロー

```javascript
async function logout() {
  // 1. Supabase Auth セッション終了
  await supabase.auth.signOut();

  // 2. リクエストヘッダーを anon キーに戻す
  H.Authorization = 'Bearer ' + SB_KEY;

  // 3. アプリ状態を初期化
  S = { inv:[], ib:[], ob:[], clients:[], scanN:0, scanMode:'入庫', curScan:null, editId:null };

  // 4. ログイン画面を表示
  showLoginScreen();
}
```

### ユーザー体験

| タイミング | 挙動 |
|----------|------|
| ログアウトボタン押下 | `確認ダイアログなし` → 即時ログアウト → ログイン画面へ |
| ブラウザタブを閉じる | セッションは `localStorage` に残るため次回アクセス時に自動ログイン |
| ブラウザのプライベートモード | セッションは保存されないため毎回ログインが必要 |
| PC共有環境 | ログアウトボタンを必ず押すよう運用ルールで周知する |

## 10. セッション切れ時の挙動

### セッション有効期間

| 種別 | デフォルト値 | 説明 |
|------|------------|------|
| `access_token` 有効期間 | 1時間 | API リクエストに使用するJWT |
| `refresh_token` 有効期間 | 60日 | access_token を更新するためのトークン |
| 自動更新 | SDK が自動実行 | アクティブ利用中は SDK が期限前に更新 |

### セッション切れ検出

```javascript
// sb.get / insert / update / del の共通エラーハンドラで検出
async function handleAuthError(response) {
  if (response.status === 401) {
    // セッション切れ → ログイン画面に遷移
    toast('セッションが切れました。再度ログインしてください', 'e');
    setTimeout(() => showLoginScreen(), 1500);
    return true;
  }
  return false;
}
```

### ユーザー体験

| 状況 | 挙動 |
|------|------|
| 1時間以内のアクティブ操作 | SDK が自動で access_token を更新（ユーザーは気づかない） |
| 1時間以上放置 → 操作再開 | SDK が refresh_token で自動復元（シームレス） |
| 60日以上未使用 | 次回アクセス時にログイン画面が表示される |
| refresh_token が無効化された | ログイン画面を表示 + `セッションが切れました` メッセージ |
| スキャン作業中に切れた場合 | 操作失敗のトーストを表示 → ログイン後に前の画面から再開可能 |

## 11. RLS 導入前の運用可否

### 結論: **Step 2 単体での運用は可能だが、セキュリティは限定的**

| 項目 | RLS なし（Step 2 完了時）の状態 |
|------|-------------------------------|
| ログイン必須か | ✓ アプリ経由では必須 |
| DB 直接アクセス | ✗ anon キーがわかれば誰でも API 直叩き可能 |
| ロール別アクセス制御 | △ アプリ UI 上の制御のみ（DBレベルは未施行） |
| データ盗聴リスク | ✗ HTTPS は有効のため通信経路は安全 |
| 退職者の不正アクセス | △ アカウント削除でアプリ経由は防げるが anon key は残る |

### Step 2 後のリスク許容判断

```
リスクレベル: 中（社内イントラネット運用の場合は低）

- 社内ネットワーク限定運用: anon key 流出リスクが低く、Step 5 まで許容可能
- インターネット公開: 早期に Step 5（RLS）まで実装すること推奨
- 現状（認証なし）: リスクレベル: 高
- Step 2 完了後: リスクレベル: 中（改善済み）
- Step 5 完了後: リスクレベル: 低
```

## 12. 移行リスクと対策

### リスク一覧

| リスク | 影響度 | 発生確率 | 対策 |
|--------|--------|--------|------|
| **全員ロックアウト** アカウント未作成で認証必須化した場合、誰もログインできなくなる | 高 | 高（対策しないと必発） | アカウント作成が完了した後に認証必須化をONにする。テスト環境で先行確認 |
| **ブラウザ共有PC** ログアウトせずに離席すると別人として使われる | 中 | 中 | 一定時間操作なしで自動ログアウト（30分 idle logout）の実装を推奨 |
| **PIN忘れ** 社員がPINを忘れても自己リセット手段がない | 中 | 高（運用開始直後に多発） | 管理者が Supabase Dashboard から password reset できる旨を周知。社内ヘルプ窓口を設置 |
| **Email Confirm ON のまま** `@wms.internal` に確認メールが届かずログインできない | 高 | 中 | 実装前に Dashboard で Email Confirm を OFF に設定する |
| **anon key のブラウザ露出** ファイルを開くと誰でも SB_KEY を確認できる | 中 | 高（現状も同様） | Step 5（RLS）で根本対策。Step 2 単体での解消は困難 |
| **カメラスキャン中のセッション切れ** スキャン作業中に401エラーが発生し在庫更新が失敗 | 中 | 低 | SDK の自動更新で通常は発生しない。エラーハンドラで適切なメッセージを表示 |
| **複数タブ操作** 同一ユーザーが複数タブでログインした場合の状態不整合 | 低 | 中 | SDK の onAuthStateChange イベントで同期。同一ユーザーの複数タブは許容 |

### デプロイ手順（推奨）

```
Phase A（準備）:
  □ Email Confirm を OFF に設定（Supabase Dashboard）
  □ 管理者アカウント作成 + テスト環境でログイン確認
  □ 全スタッフアカウント作成
  □ 各スタッフに社員番号と初期PINを通知

Phase B（本番適用）:
  □ 認証コードを本番にデプロイ（ログイン画面が出るだけ、データは全員が見える状態）
  □ 全スタッフにログインテストを依頼（1日）
  □ 問題なければ運用開始

Phase C（確認）:
  □ ログイン成功率のモニタリング（login_attempts テーブル参照）
  □ ロックアカウントがあれば管理者が解除
```
