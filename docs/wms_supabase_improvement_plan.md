## 第3章：100名規模運用時のリスク

### 3-1. 同時アクセスによる競合（最重要リスク）

```
【現状コード】complIB 関数（L542〜559）

1. sb.update('inbound', id, {status:'完了'})    ← 完了にする
2. sb.get('inventory', '&barcode=eq.'+jan)      ← 在庫を読む
3. sb.update('inventory', id, {qty: prev+qty})  ← 在庫を書く

【リスク】
ユーザーAが「2」で qty=100 を読む
ユーザーBが「2」で qty=100 を読む
ユーザーAが「3」で 100+50=150 を書く
ユーザーBが「3」で 100+30=130 を書く → 実際は 180 であるべきが 130 になる
```

100名が同時入庫完了操作を行うと、**在庫数が正しく積み上がらない**。

### 3-2. 全件取得によるパフォーマンス劣化

```
【現状コード】
renderDash（L489）: await Promise.all([sb.get('inventory'), sb.get('inbound'), sb.get('outbound')])
renderInv（L575）:  S.inv = await sb.get('inventory')  ← 毎回全件
renderMaster（L589）: S.inv = await sb.get('inventory')  ← 毎回全件
```

- 商品1,000件・在庫5,000行時に 100名が同時にダッシュボードを開くと **100回 × 3テーブル = 300回の全件取得** が発生
- Supabase 無料プランのリクエスト上限（500,000 req/月）を超過する可能性
- ブラウザ側もレスポンス待ちが重なり画面が固まる

### 3-3. anon キー露出

```
【現状コード L460〜461】
const SB_URL='https://fpobnehdqamuqlepfkrf.supabase.co';
const SB_KEY='eyJhbGci...（公開状態）';
```

- HTML ソースから誰でも Supabase URL + anon キーを取得可能
- RLS（Row Level Security）が無効のため、**外部から curl 1本で全データ取得・書き換えが可能**
- GitHub Pages は公開リポジトリの場合、ソースが全世界に公開される

### 3-4. 未登録商品での在庫登録

```
【現状コード L556〜558】complIB の else 分岐
await sb.insert('inventory', {sku:jan||ib.sku||'', ...})
→ JANコードで商品が見つからない場合に「その場で在庫登録」
```

- 誤った JAN コードでも在庫が生成される
- 重複 SKU が生まれ、在庫計算が狂う
- 100名運用では誤操作が頻発する

## 第4章：セキュリティ上の問題

### 4-1. XSS（クロスサイトスクリプティング）

wms_supabase.html は DB から取得したデータを **エスケープせず** `innerHTML` に直接挿入している。

```javascript
// 問題箇所（L539）
'<td>'+r.name+'</td>'                    // エスケープなし
'<td class="hm">'+r.client+'</td>'       // エスケープなし

// 問題箇所（L533）
'<div style="...">'+c.name.slice(0,2)+'</div>'  // エスケープなし
```

**攻撃シナリオ**: DB に `<img src=x onerror="alert(document.cookie)">` という商品名を登録
→ 全ユーザーの画面でスクリプト実行

100名運用では悪意あるユーザーによる攻撃リスクが高まる。

### 4-2. 認証なし・RLS なし

| 問題 | 影響 |
|------|------|
| ログイン不要でアクセス可能 | URL を知れば社外からも全データ閲覧・操作可能 |
| anon キーで全 CRUD が可能 | RLS ポリシーが存在しないため、Supabase API で直接削除可能 |
| 誰が操作したか記録されない | 不正操作の追跡が不可能 |

### 4-3. 入力バリデーション不足

```javascript
// 問題箇所（L719〜722）submitOB
const inv = S.inv.find(i => i.barcode === sku || i.sku === sku);
if (inv && inv.qty < qty) { toast('在庫不足', 'e'); return; }
// → S.inv が古いキャッシュの場合、在庫不足チェックが機能しない
```

- クライアントサイドの在庫チェックは、古いキャッシュデータに基づく場合がある
- 数量に負の値を入力できる可能性（`type="number"` はあるが `min` チェックがサーバー側にない）

## 第5章：データ整合性の問題

### 5-1. 非アトミックな在庫操作（再掲・詳細）

`complIB`（入庫完了）・`complOB`（出庫完了）・`processScan`（スキャン処理）の3箇所すべてで、
**読み取り → 計算 → 書き込み** が非アトミックに行われている。

PostgreSQL トランザクションの外側で実行されるため、複数ユーザーの同時操作で在庫数が不正確になる。

### 5-2. 商品マスタと在庫の混在

```javascript
// 問題箇所（L736〜742）submitMaster
// 商品マスタ登録が inventory テーブルへの insert
await sb.insert('inventory', {...payload, qty: 0});
```

- 商品情報と在庫情報が同一テーブルに格納
- 商品を削除すると在庫履歴も消える（物理削除 L624）
- 同一 JAN コードの商品を複数ロケーションに置けない（テーブル構造上）

### 5-3. 出庫時の在庫マイナス可能性

```javascript
// 問題箇所（L570）complOB
await sb.update('inventory', iv[0].id, {qty: Math.max(0, (iv[0].qty||0) - r.qty)});
```

- `Math.max(0, ...)` でマイナスを防いでいるが、**実際に在庫が足りなくても出荷完了になる**
- 複数の出庫指示が同一在庫に対して発行された場合（ロックなし）、二重出庫が起きる

### 5-4. ロット・期限管理の形骸化

```javascript
// 問題箇所（L687）submitIB
const lot = document.getElementById('ib_lot')?.value || date.replace(/-/g,'');
// → lot は入荷日の数字列（例: 20250401）で自動設定
// → expiry（賞味期限）は入力フィールドが存在しない
```

- 実際の製造ロット番号と入荷日が混在
- 期限管理が必要な商品でも expiry が空のまま登録される

## 第6章：パフォーマンス問題

### 6-1. 全件取得パターン（コード実証）

| 関数 | テーブル | クエリ | 問題 |
|------|---------|--------|------|
| `renderDash` L489 | inventory + inbound + outbound | `order=created_at.desc` のみ、LIMIT なし | 全件取得を毎回実行 |
| `renderInv` L575 | inventory | 全件取得後クライアントフィルタ | 絞り込みがサーバー側でない |
| `renderMaster` L589 | inventory | 全件取得後クライアントフィルタ | 毎ページ遷移で全件取得 |
| `renderIB` L538 | inbound | `order=created_at.desc` のみ | 全入庫履歴取得 |
| `renderOB` L563 | outbound | `order=created_at.desc` のみ | 全出庫履歴取得 |

### 6-2. 検索のたびに API 呼び出し

```javascript
// 問題箇所（L660〜664）onJan — 入庫スリップのJAN入力
async function onJan(i, jan) {
  if (!jan || jan.length < 6) return;
  if (ibCache[jan]) { fillRow(i, ibCache[jan]); return; }  // キャッシュあり
  const r = await sb.get('inventory', '&barcode=eq.'+encodeURIComponent(jan)+'&limit=1');
```

- キャッシュ機構（`ibCache`）はあるが、ページ更新で消える
- 在庫検索は `barcode=eq.` で絞り込み済みなので、これ自体は比較的効率的

### 6-3. ダッシュボード集計がクライアント側

```javascript
// 問題箇所（L494〜503）renderDash
const low = inv.filter(i => i.qty > 0 && i.qty <= (i.min_stock || 10));
const exp = inv.filter(i => i.expiry && ...);
const ibT = ib.filter(r => (r.date||'').slice(0,10) === today);
// → 全件をブラウザに読み込んでから集計
```

- 在庫5,000行 × 100ユーザー同時アクセス → 500,000行分のデータをブラウザに転送

### 6-4. 検索にデバウンスなし

```html
<!-- 問題箇所（L332）invSearch -->
<input oninput="filterInv(this.value)">
<!-- → キーを押すたびにフィルタが走る（クライアント処理なので軽いが、API版にすると問題になる） -->
```

## Phase 2（推奨）— 多ユーザー運用と業務精度向上

### P2-1. 商品マスタと在庫テーブルの分離

**目的**: 「商品の情報」と「在庫の数量・場所」を分けて正確に管理する

**改善内容**:

- 商品マスタ画面は `products` テーブルを参照するよう変更
- 在庫一覧は `inventory JOIN products` のビューを使用
- 商品削除は `deleted_at` による論理削除に変更

**対象ファイル**:
- `supabase/migrations/` に新規 SQL
- `wms_supabase.html`（`renderMaster`, `submitMaster`, `delMaster`, `renderInv` を改修）

### P2-3. 入庫・出庫ワークフローの強化

**目的**: 受け取り → 検品 → 棚入れ の工程を分業できるようにする

**改善内容**:
- **入庫ステータス**: `受付待ち` → `入荷済み` → `完了`
  - 入荷確認ボタン（数量確認）と棚入れ完了ボタンを分離
  - 棚入れ時にロケーションと実数量を入力
  - 実数量と予定数量の差異表示
- **出庫ステータス**: `ピッキング待ち` → `ピッキング中` → `出荷済み`
  - FIFO（期限の早い在庫から引当）でピッキング推奨在庫を表示
  - ピッキング完了時に在庫から減算
- **在庫ロック**: 出庫指示作成時に `locked_qty` を加算、出荷完了時に実数を減算

**対象ファイル**:
- `supabase/migrations/` に新規 SQL（ステータス追加、`locked_qty` カラム追加）
- `wms_supabase.html`（`renderIB`, `renderOB`, `complIB`, `complOB` を全面改修）

**工数**: 4日

**対象ファイル**:

| 優先度 | 改善項目 | 根拠（コード） | リスク |
|--------|---------|--------------|--------|
| **必須** | 認証機能（P1-1） | L460〜468：認証なし | 外部からの無制限アクセス |
| **必須** | RLS 有効化（P1-2） | テーブル RLS 未設定 | データ漏洩・改ざん |

| **必須** | スキャン履歴 DB 保存（P1-5） | L796：DOM 追加のみ | 画面更新で履歴消失 |
