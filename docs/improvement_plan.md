# SUPEREX LogiStation 改善計画書
## 100名規模運用に向けたスケーリング計画

**作成日**: 2026-06-05
**対象システム**: SUPEREX LogiStation（WMS）
**現行構成**: Supabase + Vanilla JS / GitHub Pages

---

## 現状分析

### システム概要
| 項目 | 現状 |
|------|------|
| フロントエンド | index.html + JS 10ファイル（約3,000行）、フレームワークなし |
| バックエンド | Supabase（PostgreSQL + Auth + REST API + RLS） |
| ホスティング | GitHub Pages（静的HTML） |
| 認証方式 | 社員番号 + 5桁PIN（疑似メール `{emp}@wms.internal`） |
| ユーザー作成 | Supabase Dashboard で手動作成のみ |
| オフライン | 非対応 |
| リアルタイム同期 | なし |

### 100名運用に向けた課題

| # | 課題 | 影響度 | 現状 |
|---|------|--------|------|
| 1 | **ページネーションなし** | 高 | 在庫・伝票を毎回全件取得。100名×同時アクセスでレスポンス悪化 |
| 2 | **バーコードスキャナ不安定** | 高 | ライブカメラ読取が動作せず、写真撮影+手動入力のみ |
| 3 | **ユーザー作成が手動** | 高 | Supabase Dashboard 操作が必要。100名の登録・管理が非現実的 |
| 4 | **リアルタイム同期なし** | 中 | 複数ユーザーが同時操作時にデータ不整合の可能性 |
| 5 | **オフライン非対応** | 中 | 倉庫内の電波不安定エリアで作業不可 |
| 6 | **監査ログUIなし** | 中 | inventory_movements テーブルはあるが画面で確認不可 |
| 7 | **セッション管理が脆弱** | 中 | タイムアウトなし、複数端末ログイン制御なし |
| 8 | **レポートが CSV のみ** | 低 | 集計・分析画面がない |
| 9 | **一括インポートなし** | 低 | 商品・ロケーションを1件ずつ手入力 |
| 10 | **通知機能なし** | 低 | 残少アラートをダッシュボードでしか確認できない |

## Phase 1: 基盤整備（安定稼働の土台）

**目標**: 現行機能の安定化と、100名が同時利用しても問題ないパフォーマンス基盤を構築する

**想定期間**: 1〜2週間

### 1-1. バーコードスキャナの修復

**現状**: ライブカメラが起動しない/読み取れない。写真撮影方式も使いにくい。
**対応**:
- html5-qrcode のライブカメラモード（`Html5QrcodeScanner`）を再実装
- フォールバック順序: ライブカメラ → 写真撮影 → 手動入力
- カメラ解像度の最適化（`{ width: 1280, height: 720 }`, `focusMode: 'continuous'`）
- 読取成功時の音声フィードバック（`AudioContext` でビープ音）
- スキャン対象フォーマットの絞り込み（EAN_13, CODE_128 のみ）

**影響範囲**: `app.js`（スキャナ共通部分）、`inbound.js`、`outbound.js`、`inventory.js`

### 1-2. ページネーションとクエリ最適化

**現状**: `v_inventory_with_names` を `select('*')` で全件取得。
**対応**:
- Supabase の `.range()` を使ったカーソルベースページネーション
- 在庫一覧: 1ページ50件、スクロール or ページ送りボタン
- 入庫/出庫一覧: 1ページ30件
- 検索時はサーバーサイドフィルタ（`.ilike()`, `.eq()`）に変更
- ダッシュボードの集計は Supabase RPC 関数化（クライアントで全件取得して集計 → サーバーで集計して結果のみ返却）

**新規DB関数**:
```sql
-- ダッシュボード用集計 RPC
fn_dashboard_stats() → { total_qty, total_skus, today_inbound, today_outbound, alert_count }
```

**影響範囲**: `dashboard.js`、`inventory.js`、`inbound.js`、`outbound.js`、新規マイグレーション

### 1-3. ユーザー管理機能（アプリ内作成）

**現状**: Supabase Dashboard でしかユーザーを作成できない。
**対応**:
- Supabase Edge Function `create-user` を作成
  - 管理者がアプリ画面から社員番号 + 初期PIN + ロール を指定して作成
  - `supabase.auth.admin.createUser()` を使用
  - Edge Function 側で管理者権限チェック
- パスワードリセット機能（管理者が任意のユーザーの PIN を再設定）
- 初回ログイン時の PIN 変更強制（オプション）

**新規ファイル**: `supabase/functions/create-user/index.ts`
**影響範囲**: `users.js`（画面からユーザー作成が可能に）

### 1-4. セッション管理の強化

**現状**: ログイン後タイムアウトなし。ブラウザを閉じるまでセッション有効。
**対応**:
- 操作無し30分でセッション自動ログアウト（idle timeout）
- ログアウト時の確認ダイアログ
- セッション有効期限の表示（ヘッダバー）

**影響範囲**: `app.js`

### 1-5. エラーハンドリングの統一

**現状**: 各画面で `if (error) { toast(...) }` のパターンだが、ネットワークエラーやセッション切れの考慮不足。
**対応**:
- 共通エラーハンドラ `handleApiError(error, context)` を追加
- 401/403 → 自動ログアウト + 再ログイン促す
- ネットワークエラー → リトライ促すUI表示
- 楽観的ロック検出 → 「データが更新されています。再読み込みしてください」

**影響範囲**: `app.js`（共通関数追加）、全JS ファイル

## Phase 2: 運用強化（マルチユーザーと現場対応）

**目標**: 複数ユーザーの同時運用を安全に行い、倉庫現場の実運用に耐える機能を追加する

**想定期間**: 2〜3週間

### 2-1. リアルタイム同期

**現状**: 画面遷移時のみデータ取得。他ユーザーの変更が反映されない。
**対応**:
- Supabase Realtime（WebSocket）を利用
- 対象テーブル: `inventory`, `inbound_orders`, `outbound_orders`
- 他ユーザーが在庫変更 → 在庫一覧に自動反映
- 入出庫のステータス変更 → 一覧に自動反映
- 「他のユーザーがデータを更新しました」通知表示

**影響範囲**: `app.js`（Realtime 初期化）、`inventory.js`、`inbound.js`、`outbound.js`

### 2-2. PWA 化（オフライン基本対応）

**現状**: 通常のWebページ。電波不安定時に操作不可。
**対応**:
- Service Worker 追加（静的アセットのキャッシュ）
- `manifest.json` 追加（ホーム画面追加対応）
- オフライン時の在庫照会（最後の同期データを表示）
- オフラインで入庫/出庫操作をキューイング → オンライン復帰時に自動送信
- 同期状態インジケーター（ヘッダに接続状態を表示）

**新規ファイル**: `sw.js`、`manifest.json`
**影響範囲**: `index.html`、`app.js`

### 2-3. 操作履歴・監査ログ画面

**現状**: `inventory_movements` テーブルにログはあるが、画面から参照不可。
**対応**:
- 新規画面「操作履歴」を追加
- フィルタ: 日付範囲、種別（入庫/出庫/移動/棚卸調整）、商品、ユーザー
- 各行に操作者名を表示
- CSV エクスポート対応
- ダッシュボードの「最近の操作」を操作履歴ページへリンク

**新規ファイル**: `js/audit.js`
**影響範囲**: `index.html`（ページ追加）、`app.js`（ナビ追加）

### 2-4. 一括インポート機能

**現状**: 商品・ロケーション・仕入先は1件ずつ画面で登録。
**対応**:
- CSV アップロードによる一括インポート
  - 商品マスタ一括登録/更新
  - ロケーションマスタ一括登録
  - 仕入先マスタ一括登録
- バリデーション結果のプレビュー（エラー行をハイライト）
- ドライラン → 本登録の2ステップ

**影響範囲**: `products.js`、`locations.js`、`suppliers.js`

### 2-5. 入出庫ワークフロー強化

**現状**: 入庫は pending → done の2ステップ。出庫も pending → shipped の2ステップ。
**対応**:
- 入庫: pending → received（入荷確認）→ inspected（検品完了）→ done（棚入完了）
  - 各ステップで異なるユーザーが操作可能（分業対応）
- 出庫: pending → picking → inspected（検品）→ shipped
  - ピッキングリスト印刷機能
- ステータス変更時の操作者・タイムスタンプ記録
- キャンセル機能の追加（在庫巻き戻し対応）

**影響範囲**: `inbound.js`、`outbound.js`、新規マイグレーション

### 2-6. 検索・フィルタの高度化

**現状**: テキスト部分一致のみ。
**対応**:
- 複合フィルタ（AND 条件の組み合わせ）
- 日付範囲フィルタ（入出庫の期間指定検索）
- ソート機能（各列ヘッダクリックで昇順/降順）
- 検索条件の保持（ページ遷移して戻った時に前の条件を維持）
- ロケーション移動画面の在庫検索をサーバーサイドに変更

**影響範囲**: 全一覧画面

## Phase 3: 拡張機能（分析・自動化・外部連携）

**目標**: データ活用による意思決定支援と、業務の自動化・効率化を実現する

**想定期間**: 3〜4週間

### 3-1. ダッシュボード強化（分析・可視化）

**現状**: 4つのKPI + 7日間棒グラフ + アラートリストのみ。
**対応**:
- 月次/週次のトレンドグラフ（入庫量・出庫量・在庫回転率）
- 商品別の在庫推移グラフ
- 仕入先別の入庫実績サマリ
- 出荷先別の出庫実績サマリ
- ABC分析（出庫頻度による商品ランク付け）
- ゾーン別の在庫占有率マップ

**影響範囲**: `dashboard.js`（大幅拡張）、新規DB関数

### 3-2. 通知・アラートシステム

**現状**: ダッシュボード表示のみ。能動的に見ない限り気づかない。
**対応**:
- アプリ内通知センター（ベルアイコン + 未読バッジ）
- 通知テーブル（`notifications`）の追加
- 自動生成通知:
  - 在庫残少（min_stock 割れ）
  - 期限切れ接近（7日/30日前）
  - 入庫の受付待ち滞留（24時間以上）
  - 棚卸の未カウント残り
- Supabase Edge Function で定期チェック（cron）
- 将来的に LINE / メール通知も検討可

**新規ファイル**: `js/notifications.js`、新規マイグレーション、Edge Function
**影響範囲**: `index.html`、`app.js`

### 3-3. 権限の細分化

**現状**: admin / operator / viewer の3ロール固定。
**対応**:
- 画面単位のアクセス制御（例: 棚卸はできるが商品マスタは閲覧のみ）
- 倉庫/ゾーン単位の権限分離（Aゾーン担当者はAゾーンのみ操作可能）
- 承認ワークフロー（出庫指示は管理者承認後に実行可能、など）
- パーミッションテーブルの追加

**新規テーブル**: `role_permissions`
**影響範囲**: `app.js`（権限チェック強化）、RLS ポリシー更新

### 3-4. レポート強化

**現状**: CSV ダウンロードのみ。
**対応**:
- 画面上でレポートをプレビュー表示
  - 入庫実績レポート（期間別・仕入先別・商品別）
  - 出庫実績レポート（期間別・出荷先別・商品別）
  - 在庫評価レポート（原単価 × 在庫数 = 在庫金額）
  - 期限切れリスト（期限30日/60日/90日以内の在庫）
- PDF出力対応（印刷用帳票レイアウト）
- 定型レポートのスケジュール自動生成

**影響範囲**: `js/reports.js`（大幅拡張）、`stocktake.js` からレポート部分を分離

### 3-5. 外部連携（将来検討）

**検討項目**（Phase 3 末期または Phase 4 として）:
- 受注管理システムとの API 連携（出庫指示の自動取り込み）
- 発注管理との連動（在庫残少 → 自動発注提案）
- ハンディターミナル対応（専用端末からの操作）
- 複数拠点対応（warehouse_id の追加）

## 技術方針

| 方針 | 内容 |
|------|------|
| フレームワーク | 現行の Vanilla JS を維持（導入コスト回避）。Phase 3 完了後に再評価 |
| DB | Supabase PostgreSQL を継続。RPC 関数の活用を拡大 |
| ホスティング | GitHub Pages を継続。Edge Function は Supabase で |
| テスト | Phase 2 から主要 RPC 関数の pgTAP テストを導入 |

## リスクと対策

| リスク | 対策 |
|--------|------|
| Supabase 無料枠の上限 | 100名規模では Pro プラン（月$25）への移行が必要。DB サイズ・API リクエスト数を監視 |
| 同時操作による競合 | RPC 関数内で `FOR UPDATE` ロックを既に使用。Realtime 導入で UI 側も即時反映 |
| Edge Function のコールドスタート | ユーザー作成は頻度が低いため許容範囲。通知 cron は Supabase の pg_cron を検討 |
| オフラインデータの衝突 | キューイング方式 + サーバー側でのバリデーションで対処。衝突時はユーザーに選択させる |
| GitHub Pages のレート制限 | 静的ファイルのみなので100名規模は問題なし。CDN キャッシュが効く |
