## 1. 現状分析

### 1-1. locations テーブル定義

```sql
create table if not exists locations (
  id                uuid primary key default gen_random_uuid(),
  code              text not null unique,      -- ロケーションコード
  zone              text not null,             -- ゾーン（A, B, C...）
  aisle             text,                      -- 通路番号（01, 02...）
  rack              text,                      -- 棚番号（01, 02...）
  level             text,                      -- 段（1, 2, 3...）
  bin               text,                      -- ビン（A, B...）
  storage_condition text,                      -- ambient/refrigerated/frozen/hazard
  capacity          int,                       -- ロケーション容量（NULLable）
  pick_priority     int not null default 100,  -- ピッキング優先度
  is_active         bool not null default true,
  created_at        timestamptz,
  updated_at        timestamptz
);
```

**インデックス**: `idx_locations_path ON (zone, aisle, rack, level, bin)`  
**RLS**: 閲覧=全認証ユーザー / 編集=admin のみ

### 1-2. location_code 命名規則の解析

モーダルのプレースホルダーから推定されるフォーマット:

```
A-01-01-1-A
│  │  │ │ └── bin（ビン）
│  │  │ └──── level（段）
│  │  └────── rack（棚番号）
│  └───────── aisle（通路番号）
└──────────── zone（ゾーン/エリア）
```

**階層構造**:
```
Zone（エリア）
  └── Aisle（通路）
        └── Rack（棚）
              └── Level（段）
                    └── Bin（ビン）
```

**実際の運用パターン**（`zoneBadge()` 関数より）:

| ゾーン | バッジ色 | 想定用途 |
|--------|---------|---------|
| A | 緑 | 常温メインエリア |
| B | 青 | 冷蔵エリア |
| C | 黄 | 冷凍エリア |
| D | 赤 | 危険物エリア |
| E | グレー | 予備/バッファ |

### 1-3. 現在のロケーション利用状況

| 機能 | テーブル | 使用カラム | 用途 |
|------|---------|-----------|------|
| 在庫管理 | `inventory` | `location_id` | 在庫スロット（SKU×ロット×ロケ×期限で一意） |
| 入庫 | `inbound_items` | `location_id` | 棚入れ先ロケーション |
| 出庫 | `outbound_items` | `from_location_id` | ピッキング元ロケーション |
| 移動 | `inventory_movements` | `location_id` | 移動元/先のログ |
| 棚卸 | `stocktake_items` | `location_id` | カウント対象ロケーション |
| 棚卸スコープ | `stocktakes` | `scope_zone` | ゾーン単位でスナップショット範囲を指定 |

### 1-4. 既存ビュー

`v_inventory_with_names` — inventory + products + locations を結合済み:
- `location_id`, `location_code`, `zone`, `storage_condition` が取得可能
- `qty`, `locked_qty`, `available_qty` で在庫量を把握可能

### 2-2. グリッド座標マッピング

locations テーブルの5階層を2次元グリッドに変換する:

| 軸 | マッピング元 | 説明 |

| X軸（列） | `aisle`（通路） | 左から右へ通路番号順 |
| Y軸（行） | `rack`（棚） | 上から下へ棚番号順 |
| セル内分割 | `level` × `bin` | セルクリック時に詳細表示 |

**ゾーン選択**: タブまたはドロップダウンでゾーンを切替え。1画面に1ゾーンを表示。

### 2-3. セル色分けルール

| 状態 | 色 | 条件 |
|------|-----|------|
| 空き | `--surface2`（薄いグレー） | そのロケーションに在庫行が0 |
| 使用中（低密度） | `--accent-light`（薄い青） | 在庫あり、capacity未設定 or 充填率 < 50% |
| 使用中（中密度） | `--accent`（青） | 充填率 50%〜80% |
| 高密度 | `--yellow`（黄） | 充填率 80%〜99% |
| 満杯 | `--red`（赤） | 充填率 100%（qty >= capacity） |
| 無効 | `--border`（ボーダー色） | `is_active = false` |

**充填率の計算**:
```
充填率 = SUM(inventory.qty WHERE location_id = X) / locations.capacity
```

`capacity` が NULL のロケーションは:
- 在庫0 → 空き
- 在庫あり → 使用中（低密度）固定

## 3. 実装方式の評価

### A案: 既存テーブルのみ利用（クライアント集計）

**方式**: `locations` + `inventory` を別々にSELECTし、JSで集計。

```js
const { data: locs } = await sb.from('locations')
  .select('*').eq('zone', selectedZone).eq('is_active', true);
const { data: inv } = await sb.from('inventory')
  .select('location_id, qty')
  .in('location_id', locs.map(l => l.id))
  .gt('qty', 0);
// JS側で location_id ごとに SUM(qty)
```

| 項目 | 評価 |
|------|------|
| 実装工数 | 低 |
| DB変更 | なし |
| リアルタイム性 | 画面表示時に毎回取得 |
| パフォーマンス | ロケ100〜500で問題なし。1000以上は注意 |
| 保守性 | シンプル |

### B案: 集計ビューを追加（DB集計）

**方式**: `v_location_summary` ビューを作成し、DB側で集計済みデータを返す。

```sql
CREATE VIEW v_location_summary AS
SELECT
  l.id, l.code, l.zone, l.aisle, l.rack, l.level, l.bin,
  l.storage_condition, l.capacity, l.is_active,
  COALESCE(SUM(i.qty), 0) AS total_qty,
  COUNT(DISTINCT i.product_id) AS product_count,
  CASE
    WHEN l.capacity IS NOT NULL AND l.capacity > 0
      THEN ROUND(COALESCE(SUM(i.qty), 0)::numeric / l.capacity * 100)
    ELSE NULL
  END AS fill_pct
FROM locations l
LEFT JOIN inventory i ON i.location_id = l.id AND i.qty > 0
GROUP BY l.id;
```

| 項目 | 評価 |
|------|------|
| 実装工数 | 中（ビュー追加） |
| DB変更 | ビュー1つ追加（テーブル変更なし） |
| リアルタイム性 | ビューなのでリアルタイム |
| パフォーマンス | 5000ロケでも高速（DB側でGROUP BY、インデックス活用） |
| 保守性 | 集計ロジックがDBに集約されて一貫性が高い |

### 採用方針: **B案（集計ビュー追加）**

**理由**:
1. 100名規模運用でロケーション1000以上を想定すると、クライアント集計はメモリ・通信量が課題
2. ビューは既存テーブルに変更を加えない（テーブル追加なし）
3. 集計ロジックがDB側にあることで、将来APIやダッシュボードからも同じデータを利用可能
4. `v_inventory_with_names` と同様のパターンで既存設計と一貫性がある

## 4. 性能評価

### 4-1. クエリ負荷シミュレーション

B案のビュー `v_location_summary` の性能:

| ロケーション数 | inventory行数（想定） | GROUP BY性能 | 通信量 | 評価 |
|--------------|--------------------|-----------|----|------|
| 100 | 〜500 | < 10ms | 〜5KB | 問題なし |
| 500 | 〜2,500 | < 30ms | 〜25KB | 問題なし |
| 1,000 | 〜5,000 | < 80ms | 〜50KB | 問題なし |
| 5,000 | 〜25,000 | < 300ms | 〜250KB | 注意（ゾーン絞り込み必須） |

### 4-2. 最適化戦略

1. **ゾーン単位で取得**: `.eq('zone', selectedZone)` でフィルタ（`idx_locations_path` が効く）
2. **1ゾーン最大表示**: 通路10×棚10=100セル程度を上限とし、DOM負荷を抑制
3. **キャッシュ不要**: ビューは毎回最新値を返すので、画面切替時に再取得するだけで十分
4. **遅延ロード**: マップタブ切替時にのみデータ取得（一覧タブ表示中は取得しない）

### 4-3. 5000ロケーション時の対策

- ゾーン数が5以上の場合、1ゾーンあたり1000ロケ以下になるため問題なし
- 1ゾーン内が100セルを超える場合は通路単位でページネーション（将来対応）
- P2段階では「1ゾーンの通路×棚を全セル表示」で十分（通常10×10=100セル以内）

## 9. DB変更一覧

| 種別 | オブジェクト | 内容 |
|------|------------|------|
| CREATE VIEW | `v_location_summary` | ロケーション別在庫集計ビュー |

**既存テーブルへの影響**: なし  
**既存ビューへの影響**: なし  
**RLS**: ビューは基テーブルのRLSを継承するため追加設定不要

### ビュー定義

```sql
CREATE OR REPLACE VIEW v_location_summary AS
SELECT
  l.id,
  l.code,
  l.zone,
  l.aisle,
  l.rack,
  l.level,
  l.bin,
  l.storage_condition,
  l.capacity,
  l.pick_priority,
  l.is_active,
  COALESCE(SUM(i.qty), 0)::int AS total_qty,
  COALESCE(SUM(i.locked_qty), 0)::int AS locked_qty,
  COUNT(DISTINCT i.product_id)::int AS product_count,
  CASE
    WHEN l.capacity IS NOT NULL AND l.capacity > 0
      THEN LEAST(ROUND(COALESCE(SUM(i.qty), 0)::numeric / l.capacity * 100), 100)::int
    ELSE NULL
  END AS fill_pct
FROM locations l
LEFT JOIN inventory i ON i.location_id = l.id AND i.qty > 0
GROUP BY l.id;
```

## 11. リスク評価

| リスク | 影響度 | 対策 |
|--------|--------|------|
| aisle/rack が NULL のロケーションが存在 | 中 | NULL の場合「未分類」グループにまとめて表示 |
| capacity が大半で未設定 | 低 | 充填率計算不可のセルは在庫有無のみで2色表示 |
| ゾーンが1つしかない | なし | ゾーン選択UIは表示するが、1つしかなければ自動選択 |
| 大量のロケーション（5000+） | 低 | ゾーン絞り込みでクエリ対象を制限。1画面100セル以内 |
