## 1. XSS 対策状況（index.html 側）

### 結論: **不十分 — 修正必須箇所あり**

#### CRITICAL（即時修正）

| # | ファイル | 行 | 問題 | 現在のコード | 修正方法 |
|---|---------|-----|------|-------------|---------|
| 1 | `app.js` | L254 | toast() の msg が未エスケープ | `` `<span>${msg}</span>` `` | `` `<span>${esc(msg)}</span>` `` |
| 2 | `app.js` | L237 | openModal() の title が未エスケープ | `` `<div class="modal-title">${title}</div>` `` | `` `<div class="modal-title">${esc(title)}</div>` `` |

#### MEDIUM（修正推奨）

| # | ファイル | 行 | 問題 | 修正方法 |
|---|---------|-----|------|---------|
| 3 | `move.js` | L92 | `i.lot_no` が未エスケープ（同じ行の他フィールドは esc() 済み） | `esc(i.lot_no)` を適用 |

#### LOW（UUID のため実害なし、防御的に修正推奨）

onclick 属性に DB の UUID を直接埋め込んでいる箇所が多数あるが、UUID は `[0-9a-f-]` のみで構成されるため XSS リスクは実質ゼロ。ただし一貫性のため将来的に `data-id` 属性 + `addEventListener` パターンへの移行を推奨。

**対象ファイル**: `products.js`, `locations.js`, `suppliers.js`, `stocktake.js`, `inbound.js`, `outbound.js`  
**対象箇所**: 約20箇所の `onclick="fn('${id}')"` パターン

## 4. クライアント管理画面の移植工数

### 概要

wms_supabase.html の `clients` テーブル + クライアントカード UI を index.html 向けに再実装する。

### 必要作業

| 作業 | 内容 | 行数目安 |
|------|------|---------|
| DB テーブル | `clients` テーブルは wms_supabase.html 側で直接 REST 利用しているだけ。index.html 側は正規化 DB（products, locations 等）を使用しているため、`clients` テーブル定義 + RLS を migration に追加 |  20行（SQL）|
| `js/clients.js` 新規作成 | 一覧（カード or テーブル）、検索、新規登録モーダル、編集、CRUD | 120〜150行 |
| `index.html` 変更 | サイドバー・ドロワー・ボトムナビにクライアント項目追加、`page-clients` div 追加、`<script src="js/clients.js">` 追加 | 15行 |
| `app.js` 変更 | `PAGE_TITLES` に追加 | 1行 |
| 入出庫との連携 | inbound_orders / outbound_orders に client_id FK を追加するか、既存 customer フィールドを活用するか設計判断が必要 | 設計次第 |

### 工数: **約1日**（DB設計 + UI実装 + テスト）

### リスク: 低

既存テーブル・画面への影響なし。新規追加のみ。

## 5. ロケーションビジュアルマップの移植工数

### 概要

wms_supabase.html の 8×10 グリッドマップ + エリア別バーチャートを locations 画面に追加する。

### 必要作業

| 作業 | 内容 | 行数目安 |
|------|------|---------|
| `js/locations.js` 変更 | 既存の CRUD テーブルの上部にマップ表示エリアを追加。inventory データを取得してグリッドに密度表示 | 80〜100行追加 |
| CSS 追加 | `.loc-cell`, `.loc-row` 等のグリッドスタイル（wms_supabase.html から移植） | 15行 |
| ロケーションコード変換 | wms_supabase.html は `A-01` 形式（行-列）。index.html の locations テーブルは `zone/aisle/rack/level/bin` の5階層。マッピングロジックが必要 | 20行 |
| クリック詳細 | グリッドセルクリックでそのロケーションの在庫一覧を表示 | 30行 |

### 工数: **約1日**（ロケーションコード変換の設計 + UI実装）

### リスク: 中

locations テーブルの構造が wms_supabase.html と異なるため、単純コピーではなくマッピングロジックの設計が必要。`zone` をマップの「エリア」として使い、`aisle-rack` をグリッド座標にする方針を推奨。

## 6. スキャン専用画面の移植工数

### 概要

現在のポップアップオーバーレイ方式に加え、スキャン専用ページ（入庫/出庫モード切替 + スキャン履歴）を追加する。

### 必要作業

| 作業 | 内容 | 行数目安 |
|------|------|---------|
| `js/scan.js` 新規作成 | 専用画面 UI（モード切替タブ、バーコード入力、商品カード表示、数量入力、履歴テーブル）| 150〜180行 |
| `index.html` 変更 | サイドバー・ドロワー・ボトムナビに「スキャン」追加、`page-scan` div 追加、script タグ追加 | 15行 |
| `app.js` 変更 | `PAGE_TITLES` に追加 | 1行 |
| カメラ統合 | 既存の html5-qrcode ライブラリを活用。ライブカメラ方式（`Html5Qrcode.start`）で連続スキャンを実装 | 50行 |
| 在庫更新連携 | スキャン→在庫増減を RPC（`fn_inbound_putaway` 等）経由で実行 | 30行 |

### 工数: **約1.5日**（カメラ統合のデバッグ含む）

### リスク: 中〜高

過去にバーコードスキャンのカメラ実装で複数回の失敗履歴がある。ライブカメラ方式は端末依存性が高く、デバッグに時間を要する可能性がある。既存の写真撮影+手動入力方式をベースにしつつ、ライブカメラは Progressive Enhancement として追加する方針を推奨。
